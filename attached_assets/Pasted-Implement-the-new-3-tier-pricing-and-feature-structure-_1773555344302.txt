canAccess() logic:
  function canAccess(feature, company, userAddons) {
    const tierFeatures = TIER_FEATURES[company.tier] // array above
    if (tierFeatures.includes(feature)) return true
    const addonFeatures = userAddons
      .filter(a => a.is_active)
      .map(a => ADDON_FEATURE_MAP[a.addon_type])
    return addonFeatures.includes(feature)
  }

PART 3 — INACTIVE USER BILLING
═══════════════════════════════════════════════


USER COUNT FUNCTION:
  getActiveUserCount(company_id):
    SELECT COUNT(*) FROM users
    WHERE company_id = $1 AND status = 'active'
  This is the ONLY count used for billing calculations.
  Inactive, seasonal_pause, invited, deleted users are NEVER counted.


SEASONAL PAUSE CRON (BullMQ, runs daily at 6am):
  Query: SELECT id FROM users
    WHERE status = 'seasonal_pause' AND pause_until <= CURRENT_DATE
  For each result: UPDATE users SET status = 'active', pause_until = NULL
  Log reactivations to audit_log.


USER DIRECTORY (Settings > Users):
  Add to each user row: status badge (Active/Inactive/Seasonal/Invited)
  Add action buttons:
    'Deactivate' → sets status = 'inactive', deactivated_at = now()
    'Seasonal pause' → opens modal: set reactivation date → status = 'seasonal_pause'
    'Reactivate' → sets status = 'active', clears pause_until
  Show billing note: 'Inactive and seasonal users do not count toward your
    user total. Only active users are billed.'
  Show active user count vs included count in the header:
    'X of 25 included users active · Y add-on users ($Z/mo)'

PART 4 — STRIPE BILLING IMPLEMENTATION
═══════════════════════════════════════════════


SUBSCRIPTION STRUCTURE (per company):
  Each company has 1 Stripe subscription with multiple items:
  Item 1: Base plan price (Pro monthly/annual or Enterprise monthly/annual)
  Item 2: User add-on metered (Pro: $1.99/user, Ent: $1.29/user)
  Item 3+: Each active add-on as a separate subscription item


CHECKOUT FLOW (/checkout):
  Read: tier (pro|enterprise), billing (monthly|annual)
  Create Stripe Customer if not exists
  Create Stripe Subscription with:
    - Base plan price ID from env vars
    - User metered price item (quantity = 0, reports monthly)
    - Apply coupon STRIPE_COUPON_FIRST30 (50% off first invoice)
  On success: update companies.tier, redirect to /dashboard?welcome=true


ADD-ON ACTIVATION (Settings > Add-ons):
  When user toggles an add-on ON:
    POST /api/billing/addons/activate
    Add subscription item to existing Stripe subscription
    Create company_addons record: is_active = true
    Return updated canAccess() state — feature unlocks immediately
  When user toggles OFF:
    Remove subscription item from Stripe (cancel at period end)
    Set company_addons.is_active = false, deactivated_at = now()
    Feature locks at end of current billing period


BACKGROUND CHECK (pay-per-use):
  POST /api/billing/addons/background-check
  Create Stripe invoice item: $9.00
  Create addon_usage record
  Charge immediately via invoice.pay()


ONBOARDING SESSION (one-time):
  POST /api/billing/addons/onboarding
  Create Stripe payment intent: $59.00
  On success: create company_addons record (one-time, no recurring)
  Schedule onboarding session in demo_bookings table


MONTHLY USER BILLING CRON (runs on billing date):
  For each Pro company: count = getActiveUserCount() - 25
  For each Enterprise company: count = getActiveUserCount() - (50 × locations)
  If count > 0: report usage to Stripe metered price item
    stripe.subscriptionItems.createUsageRecord(itemId, { quantity: count })
  If count <= 0: report 0 (no charge)


ENTERPRISE LOCATION ADD-ON:
  Pro users purchasing multi-location add-on: $49/mo per location added
  Each location add-on = new subscription item at STRIPE_PRICE_ADDON_MULTI_LOCATION
  Enterprise: locations 4+ = $49/mo each (STRIPE_PRICE_LOCATION_ADDON)
